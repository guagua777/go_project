func WithValue(parent Context, key, val any) Context {
	if parent == nil {
		panic("cannot create context from nil parent")
	}
	if key == nil {
		panic("nil key")
	}
	if !reflectlite.TypeOf(key).Comparable() {
		panic("key is not comparable")
	}
	return &valueCtx{parent, key, val}
}




返回的是一个指针，为什么函数签名上还是值类型呢，为什么不是指针类型？
21:15
这是一个非常容易混淆的概念！关键在于：Context 不是结构体，它是接口。

函数签名

Go

func WithValue(parent Context, key, val any) Context {
    return &valueCtx{parent, key, val}
}
返回类型写的是 Context，但 Context 是一个 interface，不是 struct。

接口类型的本质
在 Go 中，接口变量内部存储的是一个二元组：


Plain Text

(type, value)
当你执行 return &valueCtx{parent, key, val} 时，返回的接口值内部实际上是：


Plain Text

(*valueCtx, 指向堆上 valueCtx 实例的地址)
指针并没有丢失！ 它被"装进"了接口这个盒子里。

为什么不写成 *valueCtx 作为返回类型？

Go

// ❌ 如果这样写：
func WithValue(parent Context, key, val any) *valueCtx {
    return &valueCtx{parent, key, val}
}
这会有严重问题：

暴露了内部实现：valueCtx 是包内的私有类型（小写开头），外部包根本无法使用 *valueCtx 这个类型，调用者连变量都声明不了。
破坏了多态性：WithValue、WithCancel、WithTimeout 各自返回不同的具体类型（*valueCtx、*cancelCtx、*timerCtx），如果返回具体类型，调用者就无法用统一的方式处理它们。返回 Context 接口，调用者只需要面对一个类型：

Go

// ✅ 统一用 Context 接口接收
ctx := context.Background()
ctx = context.WithValue(ctx, "k", "v")
ctx, cancel := context.WithCancel(ctx)
ctx = context.WithTimeout(ctx, time.Second)
这是 Go 的惯用模式：标准库中大量使用这种模式——返回接口类型，隐藏具体实现：
函数	返回接口	实际返回的具体类型
io.NewReader	io.Reader	*bytes.Reader
errors.New	error	*errors.errorString
context.WithValue	Context	*valueCtx
一句话总结
Context 接口不是"值类型"，它是一个能装指针的"盒子"。 &valueCtx{...} 是指针，返回时指针被装进了 Context 这个接口盒子里，指针本身没有任何丢失或拷贝。写 Context 而不是 *valueCtx 是为了封装内部实现 + 保持多态性。






那如果不返回指针，返回值类型，即return valueCtx{parent, key, val}，是否也是可以的呢
21:19
这个问题问得非常好！答案是：能编译通过，但功能是错的。

先看方法集（Method Set）的规则
valueCtx 的 Value 方法定义在指针接收者上：


Go

func (c *valueCtx) Value(key any) any {  // 指针接收者
    if c.key == key {
        return c.val
    }
    return value(c.Context, key)
}
Go 的方法集规则：

类型	拥有的方法
valueCtx（值类型）	嵌入的 Context 提升的方法（Deadline, Done, Err, Value）
*valueCtx（指针类型）	提升的方法 + 自己定义的 Value（覆盖提升版本）
关键区别：Value 方法的行为完全不同
返回 &valueCtx{...}（指针）时：


Go

ctx.Value("myKey")
// 调用的是 *valueCtx.Value → 先检查自己的 key，匹配则返回 val
// 找不到再委托给 parent
返回 valueCtx{...}（值）时：


Go

ctx.Value("myKey")
// 调用的是嵌入的 Context 提升的 Value → 直接调用 parent.Value("myKey")
// 自己存的 key-val 完全被忽略！
用代码验证

Go

parent := context.Background()
ctx := context.WithValue(parent, "k", "v")

// 正常情况（返回指针）
fmt.Println(ctx.Value("k"))  // 输出: v  ✅

// 如果改成返回值类型
fmt.Println(ctx.Value("k"))  // 输出: nil  ❌ 永远找不到！
为什么会这样？
因为 Value 方法定义在 *valueCtx（指针接收者）上，它不在 valueCtx 值类型的方法集中。值类型只能用嵌入 Context 提升出来的那个 Value，也就是 parent.Value(key)，自己存的 key/val 形同虚设。

总结
返回方式	编译通过？	功能正确？	原因
&valueCtx{...}	✅	✅	*valueCtx 的 Value 覆盖了提升版本
valueCtx{...}	✅	❌	值类型用提升的 Value，自定义逻辑被绕过
所以返回指针不仅是为了避免拷贝，更是因为自定义的 Value 方法只存在于指针的方法集中。这也是 Go 中"方法定义在指针接收者上"的一个典型影响——值类型和指针类型的方法集可能不同，行为也可能不同。
