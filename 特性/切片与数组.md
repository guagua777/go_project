为什么用 []Grade（切片）而不是数组？
核心原因：长度不确定
看这个 Student 结构体，每个学生的成绩数量是不确定的——有的学生可能有 3 门成绩，有的可能有 10 门。而 Go 的数组有一个致命限制：


Go

// 数组——长度是类型的一部分！
var a [3]Grade   // 类型是 [3]Grade
var b [5]Grade   // 类型是 [5]Grade
// a 和 b 是不同的类型，不能互相赋值！
如果用数组，你就必须提前确定长度，而且不同长度的数组是不同类型，完全无法通用。

而切片 []Grade 是动态长度的，可以随时 append，这才是真正需要的。

数组 vs 切片 对比
特性	数组 [n]T	切片 []T
长度	固定，是类型的一部分	动态，可增长
类型	[3]int 和 [5]int 是不同类型	[]int 统一类型
零值	元素的零值	nil
作为参数传递	值拷贝（复制整个数组）	传递底层数组的引用（轻量）
可否 append	❌ 不可以	✅ 可以
什么时候用数组？
几乎不用。 在 Go 中，数组的适用场景极其有限：

长度确定且永远不变，比如：

Go

// IPv4 地址永远是 4 字节
var ip [4]byte

// SHA256 哈希永远是 32 字节
var hash [32]byte

// 转换矩阵 3x3
var matrix [3][3]float64
作为底层数据结构，切片本身就是基于数组构建的。
什么时候用切片？
绝大多数情况。 只要长度可能变化，或者你不确定长度，就用切片：


Go

Grades    []Grade      // ✅ 成绩数量不固定
Students  []Student    // ✅ 学生数量不固定
在这个项目中，Student 的成绩数量在编译时完全无法预知，所以 []Grade 是唯一合理的选择。

一句话总结
Go 中 99% 的情况用切片。 只有当长度是固定的、且是类型的本质特征时（如 IP 地址、哈希值），才用数组。

type Students []Student —— 为什么也用切片？
这种写法的含义
这行代码定义了一个新类型：


Go

type Students []Student
Students 是基于 []Student 切片创建的自定义类型。它本质上就是一个"学生列表"，但有了自己的类型名和方法。

为什么要这样做？
看这个类型后续的使用方式——它有自己专属的方法：


Go

func (ss Students) GetByID(id int) (*Student, error) {
    for i := range ss {
        if ss[i].ID == id {
            return &ss[i], nil
        }
    }
    return nil, fmt.Errorf("Student with ID %d not found", id)
}
这就是关键原因：给切片绑定方法。

如果直接用 []Student，你没法给它加方法（Go 不允许给内置/基础类型定义方法）。但通过 type Students []Student，你就可以：


Go

// ✅ 可以绑定方法
func (ss Students) GetByID(id int) (*Student, error) { ... }
func (ss Students) Average() float32 { ... }

// ❌ 如果是 []Student，不能这样写
类比理解
写法	含义	能否添加方法
students := []Student{}	一个普通切片变量	❌
type Students []Student	给切片起个新名字	✅
就像：

"hello" 是字符串 → 不能给它加方法
type Name string → 可以给 Name 类型加方法
实际效果对比

Go

// 没有自定义类型时：
student := findStudent(students, 123)  // 需要一个独立函数

// 有自定义类型后：
student, err := students.GetByID(123)   // 直接调用方法，更面向对象、更自然
这就是 "类型 + 方法" 的 Go 式面向对象风格。

一句话总结
用切片是因为学生数量不固定；用 type ... []Student 是为了给切片挂载方法，让它变成一个有行为的"智能集合"，而不是裸的数据容器。
