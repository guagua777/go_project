1）无缓冲 channel（同步通道）
go
运行
ch := make(chan int)
发送操作会阻塞，直到有另一个 Goroutine 来接收
接收操作会阻塞，直到有另一个 Goroutine 来发送
发送与接收必须配对，否则永久阻塞

这里面的阻塞，没太理解，是一开始不能发送进去，只有当两者都连上channel后，才能发送成功吗
比如：
func main() {
	ch := make(chan int)
	go func() { ch <- 100 }() // 子协程写入
	val := <-ch               // 主协程读取
	fmt.Println(val)
}
是什么时候才发送成功的？里面的执行流程是什么
一、先回答你的核心疑问
是一开始不能发送进去，只有当两者都连上 channel 后，才能发送成功吗？
完全正确。
无缓冲通道（不带第二个容量参数）：
go
运行
ch := make(chan int) // 无缓冲，同步通道
没有缓冲区，数据无法预先存入通道。
发送 ch <- x：协程立刻卡住（阻塞），一直等到有别的协程执行接收 <-ch；
接收 <-ch：协程立刻卡住（阻塞），一直等到有别的协程执行发送 ch <- x；
只有发送方和接收方都准备就绪，两边碰头，数据才会直接从发送协程拷贝到接收协程，通道本身不存放任何数据。
二、你这段代码完整执行流程
go
运行
func main() {
	ch := make(chan int)
	go func() { ch <- 100 }() // 子协程写入
	val := <-ch               // 主协程读取
	fmt.Println(val)
}
步骤拆解
主线程创建无缓冲通道 ch。
启动子 goroutine：
子协程执行 ch <- 100，尝试往无缓冲通道发数据。
此时还没有接收方，子协程立即被阻塞，挂起休眠。
主线程继续向下执行：val := <-ch，主线程开始等待接收。
此刻：
子协程：等待发送（等着有人接收）
主线程：等待接收（等着有人发送）
双方都就位，Golang 调度器完成配对。
数据直接从子协程拷贝给主线程，发送与接收同时完成。
主线程拿到 val=100，打印结果，程序正常退出。
三、反过来写就会永久阻塞（死锁）
如果把发送写在主协程，接收开协程，并且调度时机导致主协程先执行发送：
go
运行
func main() {
	ch := make(chan int)
	ch <- 100 // 主线程发送，无缓冲，立刻阻塞，再也没有机会启动子协程
	go func() { val := <-ch }()
}
主线程卡在发送语句，协程永远无法启动，没有接收者，程序死锁。
四、一句话总结无缓冲 chan
无缓冲 channel 没有存储空间，发送和接收必须同步会面。
只有发送协程 + 接收协程双双等待在通道上时，数据才能完成传递；任意一方单独执行，都会被阻塞。
